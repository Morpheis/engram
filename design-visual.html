Architecture & Design

Mental Model

A structured knowledge graph for AI agents. Build, query, and maintain relationship maps of codebases, organizations, and systems.

The Problem

Structural Knowledge Dies Every Session

During a coding session, an agent discovers chains of dependencies — components call hooks, hooks proxy to microservices, services depend on databases. That chain vanishes when the context window closes. Next session, re-discovery from scratch.

ToolWhat It DoesGap
QMD / SearchFinds documents by keywordCan't answer "what depends on X?"
MEMORY.mdFree-text long-term notesNo structure, can't traverse
Architecture docsStatic snapshotsStale within days
ChitinBehavioral patternsNot structural knowledge
Code grepFind references in sourceCross-repo invisible, redo every session
Solution

Externalized Structural Memory

A CLI tool that lets agents build directed graphs of entities and relationships. Like the structural knowledge a human engineer carries in their head — but persistent, queryable, and shareable between agents.

Core Models

Named graphs for any domain: a codebase, an org chart, an infrastructure topology, a family tree.

Core Nodes

Typed entities: services, components, databases, people, configs. With metadata and freshness timestamps.

Core Edges

Directed, labeled relationships: calls, depends_on, proxies_to, owns. With context metadata and weights.

Freshness

Staleness Detection

Code Models Git-Anchored

Models anchor to a commit hash. Freshness check diffs the anchor against HEAD, maps changed files to nodes, and marks affected subgraphs as "needs verification".

mm check hashbranch-arch Model anchored at abc1234 (7 days ago) HEAD is now def5678 Changed files: src/hooks/useFleet.ts → affects: useFleet src/routes/fleets/FleetView.tsx → affects: FleetView 2 nodes need verification, 3 edges affected

Other Models Timestamp-Based

Non-code models track when each node/edge was last verified. Query for stale entries to decide what needs re-checking.

mm q hashbranch-team --stale --days 30 Stale nodes (not verified in 30+ days): saman (person) — last verified 45 days ago deepak (person) — last verified 42 days ago 2 stale nodes found

Interface

CLI Commands

Speed-first. Every command completes in <100ms for local operations. Batch mode for bulk creation.

Building

# Create a model mm create hashbranch-arch --type code --repo ~/Hashbranch/hashbranch-medusa-2 # Add nodes and edges one at a time mm add hashbranch-arch FleetView --type component mm link hashbranch-arch FleetView calls useFleet # Or batch after a code dive mm batch hashbranch-arch <<EOF add FleetView --type component add useFleet --type hook add fleet-rest --type service link FleetView calls useFleet link useFleet calls fleet-rest EOF

Querying

# What's connected to this node? mm q hashbranch-arch fleet-rest # What's affected if I change this? mm q hashbranch-arch --affects fleet-rest # Find the path between two nodes mm q hashbranch-arch --from FleetView --to fleet-db # What's stale? mm q hashbranch-arch --stale --days 14 # Find a node across all models mm xq fleet-rest

Roadmap

Implementation Phases

Phase 1 Core MVP

Model/Node/Edge CRUD, basic queries (neighbors, depth-N traversal), batch mode, JSON export/import. Enough to start using immediately.

Phase 2 Intelligence

Path finding (A→B), blast radius, dependency chains, git commit anchoring, staleness detection, cross-model edges.

Phase 3 Agent Integration

OpenClaw skill, heartbeat-driven freshness, sub-agent model updates from diffs, DOT/Graphviz export, QMD integration.

Phase 4 Graph Backend

Storage abstraction swap to Kuzu or equivalent. Performance benchmarks, migration tooling, advanced traversals.

Commander.js

CLI framework with subcommands, flags, and help generation. Proven ecosystem.

Decisions

Key Design Choices

Single DB file, not per-model files

Cross-model queries need all models accessible. A single models.db with model_id as partition gives both isolation (filter by model) and integration (cross-model queries). Separate files would need multiple DB connections and can't do cross-model joins.

SQLite first, graph DB planned

SQLite works today with zero dependencies. The API layer stays the same regardless of backend. We'll learn what queries we actually run before committing to a graph engine. Storage abstraction makes the swap clean.

CLI, not API

Agents interact via shell commands. A CLI works with any agent framework — OpenClaw, Claude Code, Codex, anything. An API layer can be added later if a web UI is needed.

Separate from QMD and Chitin

Different concerns. QMD = document retrieval. Chitin = behavioral patterns. Mental Model = structural relationships. They can integrate (QMD indexes model exports, Chitin informs model-building) but shouldn't merge.
